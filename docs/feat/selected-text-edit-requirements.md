# Coe 选中文本语音编辑需求

## 0. 文档角色

这份文档只定义一件事：

- `Coe` 如何在“已有选中文本”的前提下，通过语音指令改写并替换这段文本

这里先明确分工：

- 本文定义目标需求、v1 范围和安全边界
- 具体架构拆分和实现顺序后续再单独写 design / implementation plan

## 1. 背景

当前 `Coe` 的主链路是普通听写：

1. 录音
2. ASR 转写
3. LLM 做文本清理
4. 把最终文本提交到输入框，或写入剪贴板再粘贴

这条链路适合“把我刚才说的话输入进去”，但不适合另一类需求：

- 用户先在文本框里选中一段已有文字
- 再通过语音说“改得更正式一点”“缩短一点”“改成英文”“把这句写得更像 commit message”
- 系统不是追加一段新文字，而是基于选区内容和语音指令，生成新的替换文本

这和普通听写有本质区别：

- 语音内容不再只是“要上屏的正文”
- 选中的原文需要作为上下文输入给 LLM
- 输出目标不再是“当前输入位置插入”，而是“替换当前选区”

## 2. 为什么 v1 先限定在 `fcitx`

这个需求建议先只做 `runtime.mode: fcitx`，原因很直接：

1. 当前 `desktop` 路径主要是“写剪贴板 + 发送粘贴快捷键”。
2. 这条路径目前没有可靠的“读取当前应用选区文本”能力。
3. 就算勉强走“复制选区 -> 读剪贴板 -> 再粘贴替换”的低配方案，也会引入：
   - 剪贴板污染
   - 对具体应用快捷键行为的依赖
   - 终端、浏览器、富文本控件之间的不一致
4. `fcitx` 路径天然更接近输入上下文本身，更有可能安全读取选区并执行替换。

因此 v1 的结论应该明确：

- **只在 `runtime.mode: fcitx` 下支持选中文本语音编辑**
- `desktop` 模式不承诺这项能力

## 3. 目标

这个需求的目标是：

1. 用户在支持 `fcitx` input context 的文本输入区域里选中一段文本
2. 触发一次语音输入，并口述“如何修改这段文字”
3. 系统把“选中的原文”和“口述的修改指令”一起交给 LLM
4. LLM 返回替换后的文本
5. 系统把原选区替换成这段新文本

除此之外，这个功能还必须满足一个额外目标：

- **失败时要 fail closed，不要把结果误插到错误位置或错误窗口**

## 4. v1 范围

v1 只做最小可用能力：

- 只支持 `runtime.mode: fcitx`
- 只支持当前 input context 中的单段纯文本选区
- 只支持“选中已有文本，再说修改指令”的单轮交互
- 复用现有录音、ASR 和 LLM 基础设施
- LLM 输出结果为“替换文本本身”，不输出解释
- 应用结果时使用替换语义，不是普通 commit 追加语义

v1 明确不做：

- `desktop` 模式下的选区编辑
- 基于剪贴板的复制/读取/恢复兜底链路
- 富文本、样式、Markdown AST 级编辑
- 多选区、多光标
- 流式改写预览
- 对“原文局部打补丁”而不是整段替换的复杂编辑协议

## 5. 用户体验

### 5.1 基本流程

用户在一个支持 `fcitx` 的文本框里：

1. 选中一段现有文本
2. 触发 `Coe`
3. 说出修改指令，例如：
   - `改得更正式一点`
   - `缩短成一句话`
   - `翻成英文，但保留产品名`
   - `改成更像 git commit message`
4. `Coe` 完成 ASR 和 LLM 处理
5. 原选区被新文本替换

### 5.2 进入编辑模式的条件

v1 只在满足以下条件时进入“选区编辑模式”：

- 当前运行在 `runtime.mode: fcitx`
- 触发本次录音时，当前 input context 可用
- 当前 input context 中存在非空选区

如果不满足这些条件，则本次语音仍然按普通听写处理。

也就是说：

- **有选区 -> 候选进入选区编辑模式**
- **没选区 -> 保持现有普通听写行为**

### 5.3 语音内容的语义

进入选区编辑模式后，用户说的话应被理解为“修改指令”，而不是“直接上屏的文本正文”。

例如：

- 选中文本：`我们明天讨论一下这个问题`
- 语音指令：`改得更正式一点`

期望结果类似：

- `我们将于明天就此问题进行讨论。`

这里的关键语义是：

- 选中文本是“被编辑对象”
- 语音内容是“编辑意图”
- 最终输出是“替换结果”

### 5.4 输出结果形式

LLM 返回值应该是替换后的文本本身，而不是解释文字。

例如允许：

- `我们将于明天就此问题进行讨论。`

不允许：

- `可以改成：我们将于明天就此问题进行讨论。`
- `这是更正式的版本：……`

### 5.5 示例

#### 示例 1：普通改写

- 选中文本：`我们明天讨论一下这个问题`
- 语音指令：`改得更正式一点`

发给 LLM 的核心语义应接近：

```json
{
  "selected_text": "我们明天讨论一下这个问题",
  "instruction": "改得更正式一点"
}
```

期望输出类似：

- `我们将于明天就此问题进行讨论。`

最终行为应是：

- 用这句新文本替换原选区
- 不是在原位置后面追加一句

#### 示例 2：压缩表达

- 选中文本：`这个方案目前存在实现复杂、测试成本高、上线风险大的问题。`
- 语音指令：`缩短成一句更直接的话`

期望输出类似：

- `这个方案实现复杂、测试成本高、上线风险较大。`

#### 示例 3：翻译并保留专有名词

- 选中文本：`请把 Coe 的 fcitx 模式作为第一阶段方案。`
- 语音指令：`翻成英文，但保留 Coe 和 fcitx`

期望输出类似：

- `Please use Coe's fcitx mode as the phase-one solution.`

#### 示例 4：没有选区

如果用户没有选中文本，直接说：

- `今天下午三点开会`

则不进入选区编辑模式，而应保持现有普通听写行为，也就是：

- 这句话按普通听写上屏
- 不触发“基于选区改写”的 prompt

#### 示例 5：处理中切换窗口

用户在 A 文本框中选中一句话并开始录音，但在处理期间切到了 B 窗口。

如果系统已经无法确认原选区仍然属于 A 中的原始编辑目标，则 v1 应：

- 放弃本次替换
- 不把结果插入 B 窗口
- 不把它降级成普通听写继续上屏

#### 示例 6：处理中选区变化

开始录音时：

- 原选区：`明天讨论一下`

处理中用户又手动改成选中了另一段：

- 当前选区：`下周再说`

如果系统发现当前选区和最初捕获的选区不再对应，则应：

- 放弃本次替换
- 不对新的选区执行盲替换

#### 示例 7：选区存在，但语音是控制命令

用户先选中一段文本，然后说：

- `切换场景到终端`

推荐的 v1 语义应是：

- 优先把这次语音当作显式控制命令处理
- 执行场景切换
- 不把选区内容拿去做文本改写

## 6. Prompt 语义

这个功能不应继续沿用“只清理 ASR 文本”的单输入心智模型。

选区编辑模式下，LLM 至少应看到两类独立输入：

1. `selected_text`
2. `instruction`

必要时还可以带上其他上下文，例如：

- 当前场景（`general` / `terminal`）
- 当前语言或 prompt 变体

但核心语义必须明确：

- `selected_text` 是要改写的原文
- `instruction` 是用户说出的修改要求
- 返回值是最终替换文本

## 7. 安全约束

这部分是 v1 最重要的要求。

### 7.1 必须绑定到原始编辑目标

普通听写允许把结果提交到“结果返回时的当前焦点 input context”。

但选区编辑不能沿用这个语义。

原因：

- 用户选中的原始文本属于某一个明确的 input context
- 如果录音或处理期间用户切到别的窗口，再把结果 commit 到“当前焦点”，就会把改写结果插到错误地方

因此选区编辑模式必须使用更强的目标绑定语义：

- **替换目标应绑定到本次录音开始时的原始选区所在 input context**

### 7.2 应用结果前需要重新验证

在真正替换文本之前，系统应重新确认：

- 原始 input context 仍然是可替换目标
- 当前待替换的选区仍然能和最初捕获的选区对应上

实现上可以用不同办法验证，例如：

- 验证 input context 身份
- 验证当前选中文本是否仍与最初快照一致
- 验证 cursor / anchor 是否仍能映射到原始选区

本文不强制具体实现细节，但强制结果语义：

- **不能在缺乏校验的情况下盲目替换**

### 7.3 校验失败时必须 fail closed

如果应用结果前发现这些情况之一：

- 原 input context 已不存在
- 焦点已切走且目标无法安全定位
- 选区已经变化
- 无法确认当前选区仍对应原始文本

则 v1 应：

- 不执行替换
- 不把结果 commit 到别的输入框
- 不把结果当作普通听写继续上屏

也就是说：

- **宁可本次编辑失败，也不要错误覆盖别处内容**

### 7.4 v1 不要求剪贴板兜底

在上述失败场景下，v1 不要求自动复制结果到剪贴板作为补救。

这是一个可以后续再讨论的增强项，但不应成为 v1 的前置条件。

## 8. 边界行为

### 8.1 没有选区

如果触发时没有选中文本：

- 不进入选区编辑模式
- 保持现有普通听写行为

### 8.2 选区为空或不可读

如果 input context 无法提供有效选区内容，或 surrounding text 不可用：

- v1 不强求在所有应用里都进入编辑模式
- 无法确认存在有效选区时，仍按普通听写处理

### 8.3 录音期间选区变化

用户可能在录音期间：

- 改变选区
- 点击别的位置
- 切到别的窗口

v1 不尝试“智能跟随”这些变化。

如果最终无法确认替换目标仍是原始选区，应直接放弃本次替换。

### 8.4 显式控制命令

现有显式控制命令，例如场景切换，仍应保留其控制语义。

推荐的 v1 规则是：

- 显式控制命令优先级高于选区编辑
- 只有当这次语音不是控制命令时，才进入普通编辑结果输出

这样可以避免“选区存在时，所有口述都被强行解释为文本编辑”。

### 8.5 空结果

如果 ASR、编辑 prompt 或最终 LLM 返回空文本：

- 不应清空原选区
- 应视为本次编辑失败

## 9. 推荐的 v1 语义

v1 推荐行为总结如下：

1. 用户触发录音时，如果当前 `fcitx` input context 有非空选区，则捕获这次编辑会话所需的原始上下文
2. 用户语音被转写为编辑指令
3. 系统把“原始选区文本 + 编辑指令”送给 LLM
4. LLM 返回替换文本
5. 应用结果前重新验证原始替换目标仍然安全可用
6. 校验通过，则替换原选区
7. 校验失败，则本次编辑失败，不做错误上屏

## 10. 验收标准

满足以下条件则认为 v1 完成：

1. 在 `runtime.mode: fcitx` 下，文本框中存在非空选区时，用户可以通过语音指令改写并替换这段文本
2. 没有选区时，现有普通听写行为不受影响
3. 选区编辑模式下，LLM 接收到的是“原文 + 指令”，而不是单一字符串清理请求
4. 结果输出是替换文本本身，不包含解释性前后缀
5. 如果录音或处理期间目标选区失效，系统不会把结果错误插入到其他输入位置
6. `desktop` 模式不会被错误宣传为已支持该功能
