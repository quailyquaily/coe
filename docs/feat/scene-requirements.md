# Coe 场景功能需求草案

## 1. 背景

当前 `Coe` 的听写链路默认只有一套校正语义。

这对普通文本输入足够，但对一些明显有上下文差异的场景不够：

- 在 terminal 里说话时，内容里可能包含 Linux 命令、路径、flag、包名、文件名
- 在普通聊天或写作场景里，这类 token 出现概率更低

因此需要给 `Coe` 增加一个运行时可切换的“场景”概念，让同一套 ASR 结果在不同场景下使用不同的校正倾向。

## 2. 目标

场景功能 v1 要解决三个问题：

1. 用户可以通过语音显式切换当前场景
2. 场景切换成功后，系统用通知明确告诉用户当前场景
3. 后续 LLM 校正可以按当前场景使用不同语义，尤其是 `terminal` 这类命令敏感场景
4. 在 GNOME 下，用户可以通过右上角托盘菜单直接查看并切换场景，以及重启 `Coe`

## 3. v1 范围

v1 只做最小可用链路：

- 引入一个 daemon 运行时状态：`current_scene`
- 至少内置两个场景：
  - `general`
  - `terminal`
- 用户通过语音命令切换场景
- 切换成功后发送桌面通知
- 场景切换命令使用一个单独的 LLM 路由步骤来决定目标场景
- 如果安装环境是 GNOME，则默认安装 `Coe` 的 GNOME 扩展
- GNOME 扩展在右上角提供一个托盘入口，用于切换场景和重启 `Coe`

v1 明确不做：

- 根据当前焦点应用自动识别场景
- 自定义场景编辑 UI
- 基于大量意图分类 prompt 的复杂 NLU
- 跨 daemon 重启的场景持久化
- “查询当前场景”的语音命令

## 4. 用户体验

### 4.1 默认状态

系统启动后默认场景为 `general`。

### 4.2 切换场景

用户说：

- `切换场景到终端`
- `switch scene to terminal`
- `シーンをターミナルに切り替え`

系统应执行：

1. 正常录音
2. 正常 ASR
3. 先走现有的 LLM correction，把原始口述整理成更稳定的文本
4. 如果修正后的文本看起来是“切换场景”指令，则进入场景切换链路
5. 场景切换成功后，发出通知，例如：
   - `Scene switched: terminal`
6. 这条语音不应再作为普通听写结果写入剪贴板、自动粘贴或提交到输入框

### 4.3 切换失败

如果门控命中了“切换场景”类前缀，但后续 LLM 路由没有解析出合法目标场景，v1 不把它视为控制命令失败，而是回退成普通听写。

行为要求：

- 保持原场景不变
- 不发“场景切换失败”通知
- 把这条文本按普通听写继续输出

### 4.4 混合语句

如果用户说：

- `切到终端，然后 ls -la`

v1 只处理“场景切换”这部分，不继续输出后半句内容。

也就是说：

- 场景成功切到 `terminal`
- 发切换成功通知
- 本次语音不再向目标应用输出 `ls -la`

### 4.5 GNOME 托盘入口

如果当前桌面环境是 GNOME，则安装流程应默认把 `Coe` 的 GNOME 扩展也安装好，不需要用户额外启用单独的安装开关。

这个扩展需要在 GNOME 顶栏右上角提供一个状态图标，点击后至少展示以下菜单结构：

```text
场景 >
重启 Coe
```

其中：

- `场景` 是一个二级菜单
- 二级菜单里列出当前所有可用场景
- 选择任意场景后，应立即把 daemon 当前场景切换到该场景
- 切换成功后，仍然需要沿用同一套“场景切换成功通知”
- `重启 Coe` 用于重启当前 `coe.service`

这个入口的目标是补充“语音切换场景”之外的一条显式控制路径，方便用户在桌面上快速确认和切换当前场景。

## 5. 场景模型

v1 的场景至少包含这些字段：

- `id`
- `display_name`
- `aliases`
- `description`
- `prompt behavior`

推荐的内置场景：

### 5.1 `general`

默认通用场景。

适合：

- 聊天
- 写作
- 普通搜索
- 非命令行输入

### 5.2 `terminal`

终端场景。

适合：

- Linux 命令
- shell 片段
- 路径
- flag
- 包名
- 文件名

这个场景下，LLM correction 需要更保守地保留命令样式 token，不要把命令误纠成自然语言。

## 6. 切换意图识别

v1 不做通用意图识别，只做一个很轻的前置门控。

### 6.1 门控时机

门控发生在现有 LLM correction 之后，也就是基于“修正后的文本”来判断。

### 6.2 门控规则

v1 可以先做成非常简单的前缀匹配：

- 中文：优先看修正后文本开头的前四个字是否属于“切换场景”类前缀
- 英文：看是否以固定短语开头，例如 `switch scene`
- 日文：看是否以固定短语开头，例如“シーン切替”或等价表达

只有命中这个门控，才进入下一步的场景路由 LLM 请求。

这意味着：

- 普通听写不会额外多走一轮场景切换判断
- 只有看起来像控制命令的文本才会进入场景切换解析

## 7. 场景路由 LLM 请求

场景切换的目标场景不靠硬编码完整解析，而是交给一个小的 LLM 路由请求。

### 7.1 输入

发给 LLM 的输入应为 JSON，并至少包含：

```json
{
  "current_scene": "general",
  "available_scenes": [
    {
      "id": "general",
      "display_name": "General",
      "aliases": ["通用", "general", "default", "普通"],
      "description": "General dictation and writing."
    },
    {
      "id": "terminal",
      "display_name": "Terminal",
      "aliases": ["终端", "terminal", "shell", "命令行", "ターミナル"],
      "description": "Linux commands, shell fragments, paths, flags, package names."
    }
  ],
  "utterance": "切换场景到终端"
}
```

这里的 `utterance` 指现有 LLM correction 之后的文本。

### 7.2 输出

要求 LLM 只输出 JSON，不要输出解释文字。

推荐最小结构：

```json
{
  "intent": "switch_scene",
  "target_scene": "terminal"
}
```

也允许扩展字段，例如：

```json
{
  "intent": "switch_scene",
  "target_scene": "terminal",
  "matched_alias": "终端"
}
```

### 7.3 解析要求

如果返回值满足以下条件，则认为切换成功：

- JSON 可解析
- `intent == "switch_scene"`
- `target_scene` 存在于当前可用场景列表

否则回退为普通听写。

## 8. 切换后的行为

切换成功后需要做三件事：

1. 更新 daemon 当前内存中的 `current_scene`
2. 发送一条通知，明确显示切换后的场景
3. 终止这次普通听写输出，不写入剪贴板，不自动粘贴

切换失败时需要：

1. 保持原场景不变
2. 不发切换失败通知
3. 把文本按普通听写正常输出

通过 GNOME 托盘菜单切换场景时，成功后的结果语义应与语音切换保持一致：

1. 更新 daemon 当前内存中的 `current_scene`
2. 发送同样的场景切换成功通知
3. 不产生任何普通听写输出

## 9. 场景对后续校正的影响

场景本质上会影响后续发给 LLM correction 的提示语义。

例如：

- `general` 更偏自然语言整理
- `terminal` 更偏命令、路径、flag、文件名保真

这部分具体如何映射到 prompt，可以由实现阶段再决定，但需求上必须保证：

- 当前场景可以影响后续的 correction 行为
- `terminal` 场景对命令行 token 的保真优先级高于 `general`
- v1 只影响 `LLM correction`
- v1 不额外影响 `ASR prompt`

## 10. 通知要求

v1 需要对“场景切换”单独发通知。

成功示例：

- 标题：`Scene switched`
- 正文：`terminal`

通知里的场景展示名需要本地化，例如：

- 中文：`终端`
- 英文：`Terminal`
- 日文：`ターミナル`

这类控制通知不应依赖“是否显示完成通知”的配置语义；它属于状态反馈，不属于普通听写完成提醒。

如果用户通过 GNOME 托盘菜单执行 `重启 Coe`，重启成功后仍应看到现有的 service ready 通知。

## 11. v1 非目标

以下内容不在 v1 范围内：

- 自动根据焦点应用切换场景
- 根据窗口标题自动切场景
- 每个应用单独记住场景
- 场景继承或分层
- 大量内置场景
- 复杂的场景参数 UI

## 12. 实现阶段需要回答的问题

这份需求文档先锁定产品行为，以下问题留给实现阶段进一步收敛：

1. 场景列表是纯内置，还是允许从配置扩展
2. 场景路由是否复用现有 correction 模型，还是走单独的轻量模型
3. `terminal` 场景如何组织额外 prompt，采用独立模板文件还是 prompt 片段拼接
4. 场景状态是否只保留在 daemon 生命周期内，还是后续支持持久化
5. 场景状态通过哪条对外接口暴露，至少要覆盖 D-Bus，后续是否也进入 `doctor` 或单独状态命令

## 13. 当前已确定的 v1 决策

- 只做两个内置场景：`general` 和 `terminal`
- 场景状态只保存在内存里，不做持久化
- 切换前缀词表先按中英日固定短语实现
- 场景路由复用现有 LLM provider / model
- v1 只让场景影响 `LLM correction`
- `terminal` 场景优先保留命令名、路径、flag、环境变量、包名、文件名、版本号
- 通知文案保持很短
- 通知中的场景展示名需要本地化
- v1 不做“查询当前场景”语音命令
- 当前场景需要对外暴露，至少进入 D-Bus 状态
